### The five changes Cooper's goal-directed process makes to conventional development
1. Design first; program second — finish interactive design before programming.
2. Separate design from programming — remove the ease-of-coding vs. ease-of-use conflict.
3. Hold designers accountable for all user-facing elements.
4. Define specific users through personas — detailed composite portraits of idealized users.
5. Work in designer/communicator pairs.

### Personas as the central tool (verbatim Cooper quotes)
- "Personas are the single most powerful design tool that we use. They are the foundation for
  all subsequent goal-directed design."
- "Until the user is precisely defined, the programmer can always imagine that he is the user."
- A persona = name, picture, job description, goals, and often a quote, on a single sheet.

### Goals vs. tasks (verbatim Cooper quotes — load-bearing for this skill)
- "Goals are not the same thing as tasks. A goal is an end condition, whereas a task is an
  intermediate process needed to achieve the goal… The goal is a steady thing. The tasks are
  transient."
- User goals (from personas) drive design decisions and tell designers which tasks are even
  worth supporting — you do not start from task analysis.

> NOTE for downstream: this goal/task split is exactly the seam between skill #7 (personas,
> goals) and skill #8 (one task model per persona-job). Goals are stable; tasks are the
> transient means — modeled later.
